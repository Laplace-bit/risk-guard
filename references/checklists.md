# Phase 1: Checklists (检查清单)

Checklists are the most evidence-based method for reducing unexpected failures.
Surgical mortality dropped 47% with a simple checklist (Gawnde). Pilots use them
before every landing. The power is in forced completeness — no known pitfall
should slip through.

## How to Use

For each item: check ✅ (confirmed safe), ⚠️ (risk exists, noted), or 🔴 (blocks execution).
Any 🔴 item MUST be resolved before proceeding.

---

## Universal Checklist (适用于所有计划)

### 基本确认

- [ ] **目标明确吗？** — 能用一句话说清成功标准吗？
- [ ] **回滚路径存在吗？** — 如果出问题，能回到之前的状态吗？
- [ ] **谁在执行？** — 每一步有明确的负责人吗？
- [ ] **时间估计靠谱吗？** — 是否用了参考类预测（同类项目的历史数据）而非内部预估？
- [ ] **依赖确认了吗？** — 所有人/服务/资源都确认可用了吗？
- [ ] **沟通到位了吗？** — 受影响的人知道这件事要发生吗？

### 常见认知陷阱

- [ ] **规划谬误检查** — 实际时间是否通常是估计的2-3倍？（如果是，调整预估）
- [ ] **锚定检查** — 时间/成本估计是否被第一个数字锚定了？（独立估算一次）
- [ ] **可用性启发检查** — 是否只因为某个风险最近发生过就高估了它？（检查基础概率）
- [ ] **确认偏误检查** — 是否只搜索了支持计划的证据？（找3个反对理由）
- [ ] **沉没成本检查** — 是否因为已经投入了就继续？（只看未来收益，不看过去投入）
- [ ] **乐观偏误检查** — "这次会不一样"有证据支持吗？

### 通信与协调

- [ ] **通知了所有受影响方吗？** — 下游依赖方、运维、用户、管理层
- [ ] **有应急联系方式吗？** — 如果出问题，第一个找谁？
- [ ] **有状态同步机制吗？** — 执行过程中怎么让相关人知道进展？

## Scenario-Specific Checklists (场景专属清单)

### 🚀 部署 (Deploy)

- [ ] **回滚方案测试过吗？** — 不仅写了，还在类似环境跑过
- [ ] **数据库迁移可逆吗？** — 特别是 schema 变更
- [ ] **功能开关就位了吗？** — 能否一键关掉新功能？
- [ ] **监控和告警设好了吗？** — 关键指标的基线是什么？
- [ ] **灰度/金丝雀发布计划有吗？** — 不是全量上线
- [ ] **依赖服务版本兼容吗？** — API 变更的消费者都更新了吗？
- [ ] **缓存策略确认了吗？** — 旧缓存会导致数据不一致吗？
- [ ] **健康检查端点更新了吗？** — 负载均衡器能正确判断新版本吗？

### 🔄 迁移 (Migration)

- [ ] **源数据完整性验证了吗？** — 迁移前做数据校验
- [ ] **双写期间数据一致性怎么保证？** — 新旧系统同时写入的策略
- [ ] **迁移脚本在大数据量下测过吗？** — 1K行能跑不代表200M行能跑
- [ ] **外键和约束都记录了吗？** — 未文档化的 FK 是定时炸弹
- [ ] **迁移失败时的回滚点在哪？** — 每一步都有回滚点，不只是整体回滚
- [ ] **停机窗口确认了吗？** — 谁批准的？多长？超时怎么办？
- [ ] **数据对比验证方案有吗？** — 迁移后怎么确认数据一致？

### 🗄️ 数据操作 (Data Operations)

- [ ] **备份存在且已验证可恢复吗？** — 不只是备份了，是恢复过
- [ ] **操作在事务中吗？** — 失败能自动回滚
- [ ] **批量操作有限流吗？** — 大批量INSERT/UPDATE是否分批+间隔
- [ ] **锁影响评估了吗？** — 是否会阻塞其他关键操作
- [ ] **数据隐私合规了吗？** — 删除/导出/脱敏是否符合法规要求
- [ ] **幂等性确认了吗？** — 重复执行不会造成数据错误
- [ ] **测试数据和生产数据隔离了吗？** — 不应该用生产数据测试

### 🔗 集成 (Integration)

- [ ] **API 契约双方确认了吗？** — 不是"文档说能行"，是实测过
- [ ] **超时和重试策略对齐了吗？** — 下游超时必须长于上游
- [ ] **限流和配额确认了吗？** — 对方API有调用频率限制吗？
- [ ] **错误处理和降级策略有吗？** — 对方挂了怎么办？
- [ ] **认证和授权配置了吗？** — Token、密钥、权限都到位了
- [ ] **网络可达性确认了吗？** — 防火墙、代理、VPN、跨环境
- [ ] **数据格式兼容吗？** — 枚举值、空值处理、时区、编码

### 📦 发布 (Release)

- [ ] **版本号和变更日志更新了吗？** — 用户能看到改了什么
- [ ] **breaking changes 通知了吗？** — API变更、配置变更、行为变更
- [ ] **文档和代码同步了吗？** — API文档、README、示例代码
- [ ] **已知问题记录了吗？** — 不影响发布但用户应该知道的限制
- [ ] **发布顺序对吗？** — 先API后消费者？先后端后前端？
- [ ] **回滚发布流程确认了吗？** — 不是回滚代码，是回滚已发布的版本
- [ ] **安全扫描通过了吗？** — 依赖漏洞、密钥泄露

### ⚙️ 配置变更 (Config Change)

- [ ] **配置变更的爆炸半径评估了吗？** — 影响范围有多大？
- [ ] **能热更新还是需要重启？** — 重启会中断什么？
- [ ] **旧值备份了吗？** — 能快速恢复吗？
- [ ] **环境一致性确认了吗？** — 所有环境（dev/staging/prod）同步确认
- [ ] **配置验证机制有吗？** — 格式错误能被拒绝而不是静默生效
- [ ] **变更审计日志有吗？** — 谁在什么时候改了什么

## Anti-Checklist (这些不是检查清单)

以下情况说明你可能在错误地使用清单：

- ❌ 用清单代替思考 — 清单确保不遗漏已知问题，不能发现未知问题
- ❌ 清单太长 — 超过30项的清单会让人跳着看，反而漏掉重点
- ❌ 只勾不查 — 每个✅都应该有证据支撑，不是"应该没问题"

**清单之后的Phase 2-6正是为了弥补清单的盲区。**
